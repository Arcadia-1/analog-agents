Meet the Agents
Four Roles. Clear Boundaries.
Each agent has strict permissions and a defined communication style. No overlap, no ambiguity.
📚

Librarian

Surveys Virtuoso libraries. Understands existing circuits. Manages cellviews.

  • Scans libraries, exports netlists, identifies topologies
  • Assesses reusability against current project specs
  • Creates/updates schematics and parameters in Virtuoso
  • Runs in background — parallel with architect
  • Uses virtuoso-bridge for all Virtuoso operations

Communication Style

"ota_fc: folded cascode, W_in=5u, Ibias=100u — reusable with resizing"
"cmp_sa uses thin-oxide — check if target process supports them"
🏗

Architect

Owns the system architecture and all testbenches.

  • Selects architecture with tradeoff analysis
  • Decomposes into sub-blocks, derives specs & budgets
  • Builds Verilog-A behavioral models for validation
  • Writes all testbenches — designer does not write testbenches
  • Defines verification plans — what to verify and how
  • Integrates verified sub-blocks at system level

Communication Style

"ENOB budget: 0.3 bit comparator noise, 0.2 bit DAC INL, 0.1 bit jitter → 7.9 available vs 7.5 required"
"PSRR testbench: AC on VDD, inputs biased at Vcm, CL=1pF on output"
🎯

Designer

Owns the netlist. Thinks in small-signal equations. Shows their math.

  • Read/write netlist files (.scs, .sp, .net)
  • Produces rationale.md with design equations
  • Can invoke optimizer when specs conflict
  • Writes custom per-block post-sim hooks
  • Never runs simulations — that's the verifier's job

Communication Style

"Sized M1 20% wider than calc minimum to account for mismatch"
"Increased Cc from 2p to 2.8p to recover 4° phase margin"
🔬

Verifier

Reviews before simulating. Never touches the netlist. Always quantifies.

  • Reviews circuit netlist AND testbench before simulating
  • Rejects bad setups — saves simulation time
  • Runs Spectre, writes margin-report.md with quantified margins
  • Routes issues: circuit problem → designer, testbench problem → architect
  • Never modifies circuit netlist or testbench

Communication Style

"phase_margin: measured 41.2°, target ≥45°, short by 3.8° at SS/125°C"
"REJECT: PSRR testbench applies AC to input, not VDD — routing to architect"

Workflow
Spec In, Silicon Out
A structured convergence loop that mirrors how real analog teams work.
S

spec.yml

Quantitative targets — gain, bandwidth, phase margin, noise, power. The contract every agent reads.

L

Librarian (background, optional)

Scan existing Virtuoso library, identify topologies, assess reusable blocks. Runs in parallel with architect.

A

Architect: Decompose

Select architecture, decompose into sub-blocks, derive specs, allocate budgets, write testbenches + verification plans.

A

Architect: Behavioral Validation

Build Verilog-A models per sub-block. System-level simulation confirms top-level specs are achievable.

D

Designer Agent

Hand-calc sizing per sub-block, produces Spectre netlist + rationale.md. Can invoke optimizer for tight specs.

V

Verifier Agent

Reviews circuit + testbench first. If both OK, runs Spectre. If not, rejects and routes feedback.

?

Convergence Check

Verifier rejected? Route to designer or architect. All specs pass? Converged. Fail? Designer revises.

↻ convergence loop (max 3 iterations per sub-block)
A

Architect: Integration

Replace behavioral models with verified netlists. Run top-level integration verification.

G

Sign-off Gate

L3 Robustness verification mandatory. All specs must pass across all PVT corners with margin.

✓

Verified Schematic

All specs pass across all corners. Verified netlist delivered. Lessons learned recorded.

Cross-Model Review

Staged Verification
Functional → Performance → Robustness
Escalate verification deliberately. The architect defines what each level checks.
L1 FUNCTIONAL

Does It Work?

Not just a DC operating point. Confirms basic function: ADC produces codes, comparator resolves, amplifier amplifies. Includes .op + .tran/.ac as needed.

TT • 27°C • .op + .tran/.ac
L2 PERFORMANCE

Does It Meet Targets?

All spec analyses at typical corner. Every result quantified with margin. Architect audits verification conditions before evaluating numbers.

TT • 27°C • Full spec analysis
L3 ROBUSTNESS

Does It Survive Corners?

Full PVT corner matrix from spec.yml. Mandatory before sign-off. No shortcuts to delivery.

TT • SS • FF • SF • FS × temp × voltage
Central Contract
One Spec Sheet. Every Agent Reads It.
Quantitative, not prose. The spec.yml is not documentation — it is the contract.
spec.yml
block: folded-cascode-ota
process: tsmc28nm
supply: 1.8V

specs:
  dc_gain:      { min: 60,  unit: dB }
  ugbw:         { min: 100, unit: MHz }
  phase_margin: { min: 45,  unit: deg }
  noise_input:  { max: 5,   unit: nV/rtHz }
  power:        { max: 1.0, unit: mW }

corners: # L3 Robustness
  - { name: tt_27c,   process: tt, temp: 27 }
  - { name: ss_125c, process: ss, temp: 125 }
  - { name: ff_m40c, process: ff, temp: -40 }
Automation
Hooks That Think For You
Simulations auto-checked. Netlist changes auto-logged. Agents write their own hooks.
PostToolUse • spectre

Auto Spec Check

After every Spectre run: parse PSF, compare against spec.yml, append to sim-log.yml, inject margin table into context. Then run designer's custom per-block hook.

PostToolUse • Write/Edit

Netlist Change Tracking

After every netlist edit: syntax check via spectre -check, then diff .param changes and record to iteration-log.yml automatically.

PreToolUse • Write/Edit

Permission Guard

Warns when a verifier agent tries to write to .scs/.sp/.net files. The netlist is the designer's territory.

Designer-defined

Custom Per-Block Hooks

Designer writes blocks/<name>/post-sim-hook.py at the start of each block. Runs after every simulation — trend plots, sensitivity tracking, whatever the designer needs.

Get Started
Five Steps to Your First Design
1
Clone the repo git clone https://github.com/Arcadia-1/analog-agents.git
2
Register the skill ln -s /path/to/analog-agents/skills/analog-pipeline ~/.claude/skills/analog-pipeline
3
(Optional) Enable cross-model review cp config/reviewers.example.yml config/reviewers.yml
# Edit with your API keys for minimax, qwen, kimi, glm
4
Configure servers cp config/servers.example.yml config/servers.yml && vi config/servers.yml
5
Create spec.yml and invoke Use the analog-pipeline skill. Design an OTA for spec.yml.
No EDA? Skip step 4. analog-agents works in review mode without servers.yml — checklists, hand-calc verification, cross-model review, and design space exploration all work without simulation infrastructure.
